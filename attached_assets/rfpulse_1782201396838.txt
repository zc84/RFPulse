Project overview Design a RFPulse  react JS web application for managing RFPs and tenders. The app supports full CRUD (Create, Read, Update, Delete) and includes role-based access. All backend interactions are mocked with static dummy data; the design must demonstrate all key UI states and flows. The overall visual style is clean, professional, and neutral (branding TBC), using a light theme with good contrast and modern typography.
Core user flows
	1	Login → email/password validation, error toasts, redirect to main list.
	2	Main list view → sortable table with filters, pagination, “Add deal” and “Refresh” buttons. Click a row to open detail page.
	3	Add deal → separate page with a form to create a new deal.
	4	Deal detail → mock detail page (read-only view) with an option to edit/delete (links or buttons).
	5	Superadmin only → user management page where superadmin can add users, set passwords, and assign permissions.
	6	Page navigation -> ability to back to the table view page

Page-by-page specifications
1. Login Page
	•	Elements:
		App logo / name placeholder (top centre)
		Email input field (with label, placeholder text “you@example.com”)
		Password input field (with show/hide toggle icon)
		“Login” button (primary, full width or centred)
	•	Validation & feedback:
		On submit, check if email exists (mocked logic: only “admin@deal.com” / “user@deal.com” are valid).
		If email not found → show a toast notification (top-right) with message: “Email not recognised. Please contact your administrator.” Toast auto-dismisses after 5 seconds or on click.
		If wrong password → toast: “Incorrect password. Please try again.”
		Empty fields → inline validation error below each field.
	•	States to design:
		Default empty form
		Inline error state (red border, helper text)
		Toast notification (success/error variants)
		Loading state on button (spinner, disabled)
2. Main List View (Authenticated User)
	•	Top bar / header:
		App name, current user name (right-aligned), logout button/icon.
		For superadmin: show an extra navigation item/link “User Management” (opens new page).
	•	Filter bar:
		Search by Name – text input with search icon, instant filter on typing (local filtering).
		Status filter – multi-select dropdown with checkboxes for: New, In Progress, Won, Lost, TBC. Apply filter button.
		“Add Deal” button – primary action, navigates to Add Deal page.
		“Refresh” button – reloads the table data (triggers a visual loading state).
	•	Table:
		Columns: ID, Name, Status, Due Date, Budget, Domain.
		All columns are sortable by clicking the header. Click toggles ascending/descending. Visual indicator (arrow icon) on the currently sorted column.
		Default sort order: primary sort by Status = “New” first, secondary sort by Due Date (earliest first).
		Pagination: below the table, show “Showing 1-10 of X deals” and page controls (Prev, Next, numbered pages). 10 deals per page.
		Each row is clickable → navigates to the Deal Detail page.
		Empty state: if no deals (after filters or empty dataset), show a centred illustration/icon and text: “No deals added yet.” with a prompt “Add your first deal” button that links to Add Deal page.
	•	States to design:
		Populated table with dummy data (at least 15 deals to show pagination).
		Empty table (no deals at all).
		No results after filtering (e.g., search “xyz” → “No deals match your criteria.”)
		Loading skeleton/spinner when “Refresh” is clicked.
		Multi-sort indicator when multiple columns sorted (optional but nice).
		Status column styled with colour badges (New=blue, In Progress=orange, Won=green, Lost=red, TBC=grey).
3. Add Deal Page
	•	Accessed via “Add Deal” button.
	•	Layout:
		Breadcrumb / back link: “← Back to deals”
		Title: “Add New Deal”
		Form fields (all required unless noted):
	▪	Deal Name (text)
	▪	Status (dropdown: New, In Progress, Won, Lost, TBC) – defaults to New
	▪	Due Date (date picker)
	▪	Budget (numeric input with currency symbol, e.g., $ or €)
	▪	Domain (dropdown: Healthcare, Fintech, Retail, Education, etc. – allow TBC values)
	▪	Additional notes / description (textarea, optional)
	▪	Upload documents section
		Action buttons: “Save Deal” (primary) and “Cancel” (secondary/outline). Cancel navigates back to list.
	•	Interactions:
		Save → on success, redirect to main list with a success toast “Deal created successfully.”
		Validation errors (missing fields) → inline error messages.
	•	State: default form, filled form, error states, loading on save.
4. Deal Detail Page (Mock)
	•	Accessible by clicking a row in the table.
	•	Layout:
		Back link: “← Back to deals”
		Heading: Deal Name with status badge next to it.
		Info cards/sections displaying:
	▪	ID, Status, Due Date, Budget, Domain, Description (if available), Documents section with list of added docs as links with ability to download.
		Action buttons: “Edit” (secondary) and “Delete” (danger, with confirmation dialog).
		Delete confirmation: modal with “Are you sure you want to delete this deal?” and “Delete” / “Cancel” buttons. On delete, redirect to list with toast “Deal deleted.”
	•	Note: Edit functionality can be a separate “Edit Deal” page (similar to Add Deal but pre-filled) – you can design either a page or an inline edit, but a separate page is simpler to mock.
5. User Management Page (Superadmin Only)
	•	Visible only when logged in as superadmin (e.g., email “admin@deal.com”).
	•	List of users in a table: Name, Email, Role (Superadmin, Editor, Viewer), Actions (Edit/Delete).
	•	“Add User” button → opens a modal or separate page:
		Name, Email, Password (with confirmation), Role dropdown (Superadmin, Editor, Viewer).
		“Create User” button.
	•	Edit user (modal or page): pre-filled fields, allow changing role, reset password.
	•	Delete user with confirmation modal.
	•	Mock data: 2-3 users with different roles.
	•	States: empty list (no other users), error toasts for failed operations.

Global design notes
Mock data & interactions
	•	All data is static mock data. No backend. Use sample deals (varied statuses, dates, domains) to populate the table.
	•	Filtering, sorting, and pagination can be simulated visually; note in annotations that these are client-side operations using the mocked dataset.
	•	Login flow: accept only the two demo credentials; all others show “contact administrator” toast.
UI components & style
	•	Colours: neutral professional palette – use a primary blue, green for success, red for errors/destructive, orange for warnings, grey for inactive. Status badges as defined.
	•	Typography: clean sans-serif (Inter, Roboto, or system font). Hierarchy: clear headings, legible table text.
	•	Spacing & responsiveness: desktop-first design (1440px wide canvas). But ensure table horizontally scrolls if needed (not strictly responsive to mobile, but consider overflow).
	•	Icons: simple, consistent line icons for actions (search, refresh, sort arrows, user, logout, etc.).
Toast notifications
	•	Position: top-right corner, slide in.
	•	Types: success (green), error (red), info (blue). Auto-dismiss after 5s, with a close button.
Loading & empty states
	•	All buttons with asynchronous actions (Login, Refresh, Save, Delete) should have a loading state.
	•	Empty tables, no search results, no users – all designed with a consistent “empty state” container.
Permissions overview for annotations
	•	Superadmin: full access, including User Management.
	•	Editor: can view, add, edit, delete deals.
	•	Viewer: can only view list and details, no add/edit/delete. UI should reflect this (hide “Add Deal”, disable edit/delete buttons). Design a note on how the viewer interface differs (e.g., disabled state for buttons).
